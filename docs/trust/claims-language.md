# Claims Language Guide

Use precise public language so developers understand RuleOak without hearing exaggerated compliance claims.

## Preferred claims

- RuleOak governs AI tool calls before execution.
- RuleOak separates policy from prompts.
- RuleOak creates evidence-backed governance records.
- RuleOak supports local-first governance workflows.
- RuleOak can require approval for risky actions.
- RuleOak can deny blocked actions at the application tool boundary.
- RuleOak provides replayable protocol artifacts for review.

## Claims that need qualification

| Claim | Safer wording |
|---|---|
| “prevents dangerous tool calls” | “prevents dangerous tool calls that are routed through RuleOak and denied by policy” |
| “audit-ready” | “audit-friendly evidence records and reports” |
| “enterprise governance” | “a path toward enterprise governance workflows” |
| “security boundary” | “application-level policy boundary with automated safety tests” |
| “compliance” | “evidence that may support internal compliance review” |

## Claims to avoid

- certified compliance;
- guaranteed regulatory compliance;
- complete sandbox;
- prevents all unsafe AI behavior;
- replaces security review;
- replaces legal review;
- automatically makes an agent safe.

## Why this matters

RuleOak’s brand should become trusted because its claims are narrow, testable, and supported by generated artifacts. Overclaiming would weaken that trust.
